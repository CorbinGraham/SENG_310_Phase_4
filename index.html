Evaluation Strategy

For our prototype that we developed in phase three, we were quite happy with the results that came through. The interface was easy-to-use and the simple layout made performing basic tasks efficient and effective. On top of that, we developed most of the functionality we wanted in our design requirements. The plan for this evaluation is to validate our opinions on the competency of the design as well as try some variations on the design to see what can be tweaked and what can be changed entirely. We knew the level of functionality would remain consistent, but by making changes in the interface (layout, colours, etc.) we can make observations during the testing that can lead us to what works the best in both variants. Also, what are the pros and cons of the design that are consistent between both prototypes? Lastly, how can we use the results we obtain to better our design and improve the prototype that we made? We also want to collect a lot of qualitative data to inform us of how to improve the user experience.

Evaluation Plan

As stated previous, the intent of our evaluation is to refine the two prototypes we have and make observations that positively influence our final product. Since our design team is in agreement with the prototype, we need other people to either validate or criticize the interfaces and go from there. Since we'll be working with about 30 university students, the opportunity to evaluate all the potential stakeholders is severely limited. However, we can still test for good usability so that any potential users won't have trouble with the design of the interface. Since all of out users in this evaluation have high computer literacy, we can theorize that the tasks each user completes are done without much hesitation. They shouldn't be intimidated by our task description or evaluation scripts, which gives us some freedom in what we can get users to do. Of course, there's time constraints but we believe we can work within the 20 minutes given to us and get a lot of useful results. Going through all the features of the product should only take a few minutes.

We don't anticipate many usability issues, however we will still get the users to try out all of the functional aspects of our prototype. That includes: using the master fader and play/pause button, turning sounds on and off, changing intensities for sounds, and adjusting volume levels. We anticipate these tasks to be relatively straightforward, but we do expect criticisms to emerge in terms of the design and user experience. We want to collect data on what the user's preferences are for each interface, that way we can create a final design that takes the best of both prototypes. We want to measure the frequency and ease at which users are able to do the basic tasks. We also want to evaluate out choice of sounds for the prototype, and collect feedback with respect to which sounds worked and which didn't. We will get users to create their own soundscapes, and to see which sounds and intensities are the most popular. Also, we'd ask users which sounds they would like added to the interface.To facilitate the evaluation process, a consent form was created for users who would be asked to try out our product.

Task Description

Participants are going to be placed in front of one of two different prototypes of the "Ambient Noise Generator". The participant will then be asked to accomplish one main task. The task is to create a soundscape using three specific ambient sounds at specific intensities and volume levels.

The user should complete the following task:

  1. Turn on the fire, waves and bird ambient noises.
  2. Set each of the sounds to a different intensity level.
  3. Have each of the sounds set to about 50% and the master volume at 75%.
  4. Attempt to export the created soundspace.
  5. The user should then reset all values.

Users will be timed on how long this task takes to complete and will be followed up with a few short questions about their experience and any other sounds that they may have wished to see. Some examples of the qualitative questions we will ask include:

  • Did the icons behave as you expected? Were the sounds similar to what you expect?
  • Are there any sounds that you felt were unnecessary? Which sounds? Which aspects of the site, if any, did you find frustrating to use?

We expect that we will collect the following quantitative results from our task description:

  • How long does it take each user to complete each task.
  • How often do users correct or repeat actions.

Results

Our group conducted our user study in the lab session on the 18th of March. We would end up only having two users participate in the study due to time constraints. The users were split into two groups each with half our team. Each of the groups would try the task described above with the two different prototypes we created but due to time constraints not all features were able to be as well discussed as we would have liked. Additionally, users did not have a chance to try both prototypes as we would have wanted them to. For the user using the new prototype version, the following qualitative results were found.

  • Waves sounds were hard to find since the icon was non-obvious.
  • Sounds overall were what was expected though.
  • It was not obvious that pressing the icons would start the sounds.
  • It additionally was confusing that the volume of each sound starts at 0 even after turning the sound on.

For the user using the old prototype version, the following qualitative results were found.

  • The user had troubles with volume slider starting at 0. The default volume should be in the middle.
  • The user perceived the river icon to be a waterfall not a river and the noise icon was not obvious to him.
  • There should be more animal sounds of different kinds and some thunder noises.
  • The user did not notice the top right hand bar at first; the top bar is hard to see compared to blue, and it can blend into the background of browsers using a dark color palette.
  • The user suggested that the background color should change as different sounds are being played to give more of a visual indication that sounds are being played.

Recommendations

In conclusion, some recommendations from the user studies that could be implemented to improve the prototype include clarifying the appearance of certain interfaces. For example, the icon looks and appearances should be updated so that it is more obvious what each sound is and the volume slider should be placed at an audible level at the start so that when users press play it starts playing immediately.

We found that the dropdown menu prototype was very easy to understand for new users, while some icons in the corresponding menu of the original prototype were difficult to understand. Switching to a dropdown menu style would improve learnability, but changing the icons to more recognizable ones could have the same effect. There are also other more general recommendations.

Some users struggled with figuring out input, so one recommendation would be to change the volume sliders to default at 50% or some level greater than 0% so users get more feedback when they first interact with the icons. In the same vein, adding on-hover interactions to the icons could give better hints that they can be interacted with. Some users found some icons not descriptive enough to determine which sounds they refer to. Potential fixes to this issue would be to change the icons, or give the icons visible titles of some form.
